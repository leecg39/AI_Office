# Issue Run Governance

## 목적

Paperclip의 issue run orchestration 원칙을 Connect AI 작업 큐와 진행률 관리에 적용한다. 목표는 한 작업이 중복 실행되거나 92% 같은 중간 상태에서 방치되는 일을 줄이는 것이다.

## 실행 소유권

- 하나의 작업에는 한 번에 한 명의 실행 소유자만 둔다.
- 같은 담당자가 같은 작업을 다시 깨우면 새 실행을 만들지 말고 기존 실행에 합친다.
- 다른 담당자의 참여가 필요하면 현재 실행이 끝난 뒤 이어서 실행되도록 다음 행동으로 남긴다.
- 병렬 처리는 같은 작업 안에서 무작정 늘리지 말고, 분리된 하위 작업일 때만 허용한다.

## 상태 종료 규칙

작업은 다음 중 하나로 반드시 닫는다.

- `done`: 결과와 검증이 완료됨
- `failed`: 실패 문자열과 재시도 조건이 기록됨
- `blocked`: 사용자 승인, 외부 로그인, 결제, 권한 등 명확한 막힘이 있음
- `cancelled`: 사용자가 중단했거나 더 이상 유효하지 않음

## 진행률 규칙

- 진행률은 실제 단계 완료와 맞아야 한다.
- 결과 생성 대기, 모델 호출, 링크 확인, 저장 확인 같은 마지막 단계는 별도 상태 메시지를 남긴다.
- 실패한 모델/API/OAuth 호출은 제공자, 모델명, HTTP 상태, 실패 문자열을 분리해 기록한다.
- 완료 결과는 완료 목록과 Brain 저장 여부를 확인한 뒤 `done`으로 닫는다.

## UI/보고 요구

- 사용자에게 보이는 오류는 원문 실패 문자열을 숨기지 않는다.
- 실행 중 상태는 짧은 메시지로 갱신한다.
- 오래 걸리는 작업은 `진행 상황`과 `다음 행동`을 남긴다.
- 승인 필요 작업은 승인 게이트로 보내고 임의 실행하지 않는다.

## 수용 기준

- 같은 작업에 활성 실행이 2개 이상 생기지 않는다.
- 같은 담당자의 반복 wake는 기존 실행으로 합쳐진다.
- 다른 담당자의 요청은 순서가 남고, 현재 실행이 끝난 뒤 처리된다.
- 작업은 `done`, `failed`, `blocked`, `cancelled` 중 하나로 종료된다.
